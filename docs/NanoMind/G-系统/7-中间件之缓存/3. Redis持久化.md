## 基础

#### 如何保证 Redis 数据不丢失？

持久化机制。

#### 有哪些持久化方式？

RDB 持久化、AOF 持久化、混合持久化。

![](/zimg/yuque-1778165492440-1.png)

#### 如何选择持久化方式？

|                |            |                      |
| -------------- | ---------- | -------------------- |
| **持久化方式** | **RDB**    | **AOF**              |
| 启动优先级     | 低         | 高                   |
| **文件体积**   | 小         | 大                   |
| **恢复速度**   | 快         | 慢(需要重放所有命令) |
| **数据安全性** | 容易丢数据 | 根据策略决定         |

生产环境可以都启用，Redis 启动时如果既有 RDB 文件又有 AOF 文件则优先选择 AOF 文件恢复数据，因为 AOF 一般来说数据更全一点。

## RDB 持久化

#### 什么是 RDB 持久化？

RDB(Redis DataBase) 持久化是**把当前进程的全量数据生成快照保存到硬盘的方式**。

#### RDB 持久化是如何实现的？基本流程是怎么样的？

RDB 持久化使用 `bgsave` 命令触发。

`bgsave` 具体流程：

- **父进程执行 fork 操作创建子进程**。这个过程中父进程是阻塞的，Redis 不能执行来自客户端的任何命令，但这个**阻塞时间很短**。不过**如果内存过大也会阻塞一段时间**。
- **父进程 fork 创建完子进程后，可以继续响应其他读写命令**。
- **子进程创建 RDB 文件，根据父进程内存快照生成临时快照文件，完成后对原有文件进行原子替换**。

#### 什么是写时复制机制？子进程持久化时父进程为什么能继续响应读写命令？

Redis 借助操作系统提供的**写时复制技术**(Copy-On-Write，COW)，在生成快照的同时，依然可以正常处理写命令。**持久化操作是由** `**bgsave**` **子进程进行的，**`**bgsave**` **子进程是由主线程** `**fork**` **生成的，所以可以共享主线程的所有内存数据。简单理解也就是基于当前进程复制了一个进程**。

`bgsave` 子进程运行后，开始读取主线程的内存数据，并把它们写入 RDB 文件。此时，如果主线程对这些数据也都是**读操作**，那么主线程和 `bgsave` 子进程**相互不影响**。但如果主线程要**修改一个数据，那这个数据就会被复制一份，生成该数据的副本**。然后 `bgsave` 子进程会把这个副本数据写入 RDB 文件，而在这个过程中，**主线程仍然可以直接修改原来的数据，也就是可以仍然正常处理写操作**。这既保证了快照的完整性，也**允许主线程同时对数据进行修改，避免了对正常业务的影响**。

![](/zimg/yuque-1778165492669-2.png)

所以快照持久化可以完全交给子进程来处理，父进程则继续处理客户端请求。

**子进程因为数据没有变化，它能看到的内存里的数据在进程产生的一瞬间就凝固了，再也不会改变，这也是为什么 RDB 持久化叫「快照」的原因**！

#### RDB 持久化方式的优缺点？

优点：RDB 文件是紧凑**压缩的二进制文件，消耗空间少**。RDB 文件代表某个时间点上的所有数据快照，所以**非常适合于备份，全量复制等场景**。Redis **加载 RDB 文件恢复数据的速度**远远快于 AOF 方式。

缺点：RDB 持久化的是内存快照，数据量较大，所以 RDB 属于**重量级操作**，不能频繁执行，因此**无法做到实时持久化，秒级持久化**。一般用于数据冷备和复制传输。

## AOF 持久化

### 基础

#### 什么是 AOF 持久化？

**AOF(Append Only File) 持久化是将 Redis 执行的每次写命令记录到单独的日志文件(appendonly.aof)中，先写入 os cache，每隔一段时间 fsync 到磁盘，当 Redis 重启时再次执行 AOF 文件中的命令来恢复数据**。

#### AOF 的流程是怎么样的？

![](/zimg/yuque-1778165492932-3.png)

整体流程：

- **命令写入缓冲区**。所有写命令都是先**追加到** `**aod_buf**`**(缓冲区)中，之后内核会根据一定的同步策略将缓冲区的命令刷到磁盘中**。
- **缓冲区输入磁盘**。缓冲区缓存的更新写入到 AOF 磁盘进行持久化。这里可以控制缓存区同步刷盘策略。
- 如果 AOF 文件过大，会进行 AOF **重写**。

#### 为什么 AOF 持久化是先修改内存数据再写 AOF 文件？

有如下好处：

- **在命令执行后才记录日志，所以不会阻塞当前的写操作，这样可以直接在内存修改数据**。性能更高。
- **只有命令能执行成功，才会被记录到日志中**，可以避免出现记录错误命令的情况。

#### AOF 日志是异步写的吗？是什么线程写的？

AOF 日志是由**读写主线程**写的。由于每次写的时候是**先写入到缓冲区，是操作的内存，因此速度也是非常快的**，基本不会出现阻塞的问题。

#### AOF 可能丢失数据吗？使用 AOF 有何风险？

主要有两个风险：

- **数据丢失风险**：如果刚执行完一个命令，还没有来得及记日志就宕机了，那么这个命令和相应的数据就有丢失的风险。
- **阻塞风险**：AOF 虽然避免了对当前命令的阻塞，但可能会给下一个操作带来阻塞风险。这是因为 AOF 日志也是在主线程中执行的，如果在把日志文件写入磁盘时，磁盘写压力大，就会导致写盘很慢，进而导致后续的操作也无法执行了。

### AOF 刷盘策略

#### 什么是 AOF 的刷盘策略？AOF 的如何控制使用的风险？

使用 AOF 持久化时，**更新命令是先写入到 aod_buf(缓冲区)中的，需要刷入磁盘才叫持久化，刷入磁盘的策略就是缓存区同步刷盘策略，由参数 appendfsync 控制**。

可选策略：

- `**always**`**(同步写回)**。命令写入 `aof_buf` 后**立即调用系统 fsync 操作同步到 AOF 文件，**`**fsync**` **完成后线程返回**。这样每次有写命令都要同步到磁盘，硬盘 IO 成为性能瓶颈，此时只能支持大约几百 TPS 写入，别用！
- `**no**`**(操作系统控制的写回)**。命令写入 `aof_buf` 后调用系统 `write` 操作，不对 AOF 文件做 `fsync` 同步；同步**由操作系统负责**，通常同步周期为 30 秒。文件同步的时间不可控，且缓冲区中堆积的数据会很多，**数据安全性无法保证**。
- `**everysec**`**(每秒写回)**。命令写入 `aof_buf` 后调用系统 `write` 操作，`write` 完成后线程返回；`**fsync**` **同步文件操作由专门的线程每秒调用一次**。它是前述两种策略的**折中**，是性能和数据安全性的平衡，因此是 Redis 的默认推荐配置。理论上来说系统宕机丢失的数据也在 1 秒内。

#### 如何选择 AOF 的刷盘策略？

三种写回策略体现了系统设计中的一个重要原则 ，即 `trade-off`，或者称为 "**取舍**"，指的就是**在性能和可靠性保证之间做取舍**。

**所以需要根据实际的业务场景进行选择取舍**。想要获得高性能，就选择 `No` 策略；如果想要得到高可靠性保证，就选择 `Always` 策略； 如果允许数据有一点丢失，又希望性能别受太大影响的话，那么就选择 `Everysec` 策略。

#### AOF 刷盘中的系统调用 write 与 fsync 是如何实现的？

`**write**` **系统调用会触发延迟写机制**。Linux 在内核提供页缓冲区来提高硬盘 IO 性能。`write` 操作在写入系统缓冲区后直接返回。同步硬盘操作依赖于系统调度机制。同步文件之前，如果系统宕机，那么缓冲区内的数据就会丢失。

`**fsync**` **系统调用针对单个文件操作(比如 AOF 文件)，将阻塞直到数据写入硬盘后返回，做强制硬盘同步**，保证数据持久化，但是开销更大。

### AOF 重写

#### 什么是 AOF 重写？

AOF 重写机制就是在重写时，**读取内存数据库中现有的最新数据，将其转换为对应写数据的命令并同步到一个新的 AOF 文件中，而不会对旧的 AOF 文件进行任何读写操作**。

#### 为什么需要进行 AOF 重写？AOF 文件过大会怎么样？

Redis 服务器执行的写命令会随时间越来越多，AOF 文件也会越来越大，过大的文件会带来一些性能问题：

- **影响写入效率**。大文件写入命令效率变低。
- **宕机后数据恢复时间变长**。恢复数据时需要把命令重新执行，时间更长，对业务影响更大。

**AOF 重写会定期重写 AOF 文件，减小了 AOF 文件的体积**。由于重写后 AOF 执行的命令减少了，文件重写既可以减少文件占用的空间，也可以加快加载恢复的速度。

#### AOF 重写流程是什么样的？

和 AOF 日志由主线程写回不同，**重写过程是由后台线程** `**bgrewriteaof**` **来完成的**，这也是为了**避免阻塞主线程**而导致数据库性能下降。

![](/zimg/yuque-1778165493245-5.png)

流程如下：

- **父进程执行** `**fork**` **操作创建子进程**。这个过程中父进程是阻塞的，但时间很短。
- 父进程 `fork` 后，父进程可以继续响应其他读写命令。之后 Redis 的所有**写命令依然写入 AOF 缓冲区，并根据** `**appendfsync**` **策略同步到硬盘，保证原有 AOF 机制的正确**。
- 由于 `fork` 操作使用写时复制技术，子进程只能共享 `fork` 操作时的内存快照数据。由于**父进程依然在响应数据更新命令**，因此 Redis 使用 AOF 重写缓冲区(图中的 `aof_rewrite_buf`)**保存这部分更新的数据**，防止新 AOF 文件生成期间丢失这部分数据。也就是说 `bgrewriteaof` 执行期间，**Redis 的写命令同时追加到** `**aof_buf**` **和** `**aof_rewirte_buf**` **两个缓冲区。缓冲区专门用于保存在重写期间操作数据带来的变化**。
- **子进程根据内存快照，按照命令合并规则写入到新的 AOF 文件**。子进程写完新的 AOF 文件后，向父进程发信号，父进程更新统计信息，具体可以通过 `info persistence` 查看。
- 父进程把 AOF 重写缓冲区(`aof_rewrite_buf`) 的数据变更写入到新的 AOF 文件，这样就保证了新 AOF 文件所保存的数据库状态和服务器当前状态一致。
- 使用新的 AOF 文件替换老文件，完成 AOF 重写。

#### 如何控制重写的频率与时机？

通过 `auto-aof-rewrite-min-size` 和 `auto-aof-rewrite-percentage` 参数配置。超过一定阈值就会触发。

#### 文件重写为什么能压缩 AOF 文件？

- **过期的数据不需要再写入文件**。
- **无效的命令不再写入文件**：如有些数据被重复设值(`set mykey v1, set mykey v2`)，有些数据被删除了(`sadd myset v1, del myset`)等。
- **多条命令可以合并为一个**：如 `sadd myset v1, sadd myset v2, sadd myset v3` 可以合并为 `sadd myset v1 v2 v3`。

## 混合持久化

#### 什么是混合持久化？

重启 Redis 时，很少使用 RDB 来恢复内存状态，因为可能会丢失数据。通常使用 AOF 日志重放，但是重放 AOF 日志性能相对 RDB 来说要慢很多，这样在 Redis 实例很大的情况下，启动需要花费很长的时间。

Redis 4.0 为了解决这些问题，新增了一个新的持久化选项：**混合持久化**。**将 RDB 文件的内容和增量的 AOF 日志文件存在一起。这里的 AOF 日志不再是全量的日志，而是自持久化开始到持久化结束的这段时间发生的增量 AOF 日志**。

![](/zimg/yuque-1778165493629-7.png)

简单来说就是**内存快照以一定的频率执行，在两次快照之间，使用 AOF 日志记录这期间的所有命令操作**。这样一来，快照不用很频繁地执行，这就避免了频繁 `fork` 对主线程的影响。而且 AOF 日志也只用记录两次快照间的操作，也就不需要记录所有操作了，因此就不会出现文件过大的情况了，也可以避免重写开销。

于是**在 Redis 重启的时候，可以先加载 RDB 的内容，然后再重放增量的 AOF 日志就可以完全替代之前的 AOF 全量文件重放，重启效率因此大幅得到提升**。

## 持久化的问题与优化 🌟

#### 持久化可能导致阻塞吗？

可能导致**阻塞**。

当 Redis 做 RDB 或 AOF 重写时，必不可少的操作就是**执行** `**fork**` **系统调用来创建子进程，但这是一个重量级操作。虽然** `**fork**` **创建的子进程不需要拷贝父进程的物理空间，但是会复制父进程的空间内存页表(页表相当于内存的索引、目录)**。所以 `fork` 操作耗时与进程总内存量息息相关。父进程的数据空间越大，内存页表越大，`fork` 时复制耗时也会越多，所以一般需要控制一下 Redis 实例的内存，不是说越大越好。

但是 `fork` 之后持久化都是由子进程执行的，主进程可以正常响应读写命令。
