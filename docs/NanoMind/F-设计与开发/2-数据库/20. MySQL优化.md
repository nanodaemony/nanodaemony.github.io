## 基础

#### 数据库优化的目标是啥？

如果站在数据库的角度，那么 SQL 优化就是为了达到两个目标。

1. **减少磁盘 IO**，这个又可以说是尽量避免全表扫描，尽量使用索引以及尽量使用覆盖索引。
2. **减少内存 CPU 消耗**，这一部分主要是尽可能减少排序，分组，去重之类的操作。

#### 有哪些数据库优化方向？优化手段？

数据库优化主要包含以下内容：

- **硬件资源优化**：换更大更强的机器。提供内存、CPU 的配置等。
- **操作系统优化**：调整操作系统的参数设置。
- **服务器/引擎优化**：也就是针对数据库软件自身进行优化，比如调整事务隔离级别。在 MySQL 里面还可以针对不同的引擎做优化，比如调整 InnoDB 引擎的日志刷盘时机。
- **SQL 优化**：针对的就是 SQL 本身了。

## 优化工具

#### 有哪些 SQL 优化工具？

常用工具：

- **Explain 关键字**。分析 SQL 的执行计划。
- **Trace 工具**。用于**分析和调试 SQL 查询的执行过程**。通过 Trace 工具，可以深入了解查询的执行计划、资源消耗、锁等待等信息，从而优化查询性能、排查问题或诊断性能瓶颈。Trace 工具输出更加详细，能比较直观的看出 SQL 成本选择的过程。

#### Explain 工具有何用处？

使用 EXPLAIN 关键字可以**模拟优化器执行 SQL 语句，并返回一个执行计划**。在 SELECT 语句之前加 EXPLAIN 关键字，MySQL 会在查询上设置一个标记，执行查询会返回执行计划的信息，而不是执行这条 SQL。可以用于分析查询语句或结构的性能瓶颈。

#### Explain 输出有哪些属性？

```
+----+-------------+-------+------------+-------+---------------+----------+---------+------+------+----------+--------------------------+
| id | select_type | table | partitions | type  | possible_keys | key      | key_len | ref  | rows | filtered | Extra                    |
+----+-------------+-------+------------+-------+---------------+----------+---------+------+------+----------+--------------------------+
|  1 | SIMPLE      | t2    | NULL       | index | NULL          | idx_abc | 198     | NULL |    5 |    20.00 | Using where; Using index |
+----+-------------+-------+------------+-------+---------------+----------+---------+------+------+----------+--------------------------+
```

Explain 会返回很多结果，重要的属性如下：

- `**type**`**：表示\*\***访问类型**，即表示 MySQL 如何查找表中的行，是较为重要的一个指标。**从最优到最差依次为：**`**system > const > eq_ref > ref > range > index > ALL**`**。range 表示使用一个索引来检索给定范围的行。index 表示扫描全索引就能拿到结果。ALL 表示全表扫描\*\*。
- `**possible_keys**`：表示本次查询**可能会使用哪些索引来进行查找**。
- `**key**`：表示查询实际采用哪个索引来进行查找。**如果没有选择使用索引，则是 NULL。**`**key = primary**` **表示使用了主键。**`**key = null**` **表示没有用到索引**。
- `**key_len**`：表示在索引里使用的字节数。
- `**rows**`：MySQL 估计要读取的行数。
- `**Extra**`：补充信息。**Using index 表示使用了覆盖索引。Using where 表示使用 WHERE 语句来处理结果，并且查询的列未被索引覆盖。Using index condition 表示使用了索引下推。Using filesort 表示使用了文件排序**。

#### 如何判断一条 SQL 走没走索引?

使用 `Explain` 输出其执行计划。首先看 key 字段有没有值，有值表示用到了索引树，但是具体是怎么用的，还得看 type 和 extra 字段的输出。

## 查询性能优化

### 通用优化思路

#### 有哪些通用的 SQL 查询优化原则？方式？思路？

一些基本的思路：

- **减少查询的数据量**。比如只查询必要的列，使用 LIMIT 限制返回的行。
- **切分大查询**。一个大查询如果一次性执行的话，可能一次锁住很多数据，导致占满整个事务日志，耗尽系统资源，阻塞很多小但重要的查询。比如分批次删除数据。
- **减少扫描行数**。使用索引减少扫描行数，使用覆盖索引减少回表查询开销。

### 查询排序优化

#### MySQL 查询后如何进行数据排序？有何排序方式？

**ORDER BY 语句可以对查询出来的数据进行排序**。进行排序时可能出现下面的排序方式：

- **索引排序(index sort)**：直接在**索引上完成了排序**。效率很高。
- **文件排序(file sort)**：索引无法排序，**数据量小的情况下，在内存中完成排序；数据量大的情况下，在内存中无法无法完成排序，会使用基于磁盘的文件排序**。效率很低。

#### 排序是否会使用索引？

**会**。所以最好对排序的字段建立索引。

#### 什么是文件排序？

当不能使用索引生成排序结果的时候，MySQL 需要自己进行排序，如果数据量小则在内存中进行，如果数据量大则需要使用磁盘排序，MySQL 将这个过程统一称为**文件排序(filesort)**，即使**完全是内存排序不需要磁盘文件时也叫文件排序。在内存中进行排序的区域叫做 "排序缓冲区(Sort Buffer，这是一块内存区域)"**。

当查询数据后，如果需要**排序的数据量小于排序缓冲区**，MySQL 将使用此内存进行 "**快速排序**" 操作。如果内存不够排序，那么 MySQL 会先将数据分块，对每个独立的块使用 "快速排序" 进行排序，并将各个块的排序结果存放在磁盘上，然后将各个排好序的块进行**合并**(merge)，最后返回排序结果。

#### 如何确定使用了文件排序？

可以使用 Explain 和 Trace 分析 SQL 语句来确定是否使用了文件排序。

- 使用 Explain 结果中会出现 "**Using filesort**" 的提示。
- 使用 trace 工具的分析结果中会有 **filesort_summary** 的信息。

#### 文件排序的算法有哪些？

文件排序算法有：

- **双路排序(又叫回表排序)**：首先根据相应的条件**仅取出需要排序的字段和数据行 ID，然后在 sort buffer 中进行排序，排序完后需要再次回表查询得到其它需要的字段**。由于只把一部分数据加载到内存排序，所以占用的内存小一点，但是排序完成需要**回表查询**。
- **单路排序**：一次性**取出满足条件行的所有字段，然后在排序缓冲区中进行排序**。由于把全部需要的数据都加载到了内存，所以占用内存较大，但是排序完成后不需要进行回表了。

#### 出现了 Using filesort 如何优化？

一些方向：

1. **尽量使用索引排序。**看看是否能优化为索引排序。
2. **优化 MySQL 配置。**还可以调整 `sort_buffer_size` 参数。这个参数决定了排序操作可以使用的内存量。增加其值可以提高处理大型排序操作的能力。

### 分页查询优化

#### 普通的分页查询优化

一个常见的分页查询：

```
SELECT * FROM employees LIMIT 10000, 10；
```

表示从表 `employees` 中取出从 10001 行开始的 10 行记录。如果没有索引，这里看似只查询了 10 条记录，实际这条 SQL 是**先读取 10010 条记录，然后抛弃前 10000 条记录，然后读到后面 10 条想要的数据**。因此要查询一张大表比较靠后的数据，执行效率是非常低的。

优化此类分页查询的一个最简单的办法就是**尽可能地使用索引覆盖扫描，而不是查询所有的列**。然后根据需要做一次关联操作再返回所需的列。**分页查询时，一定要使用主键来限定查询范围 (WHERE id > 10000 LIMIT 10) 进行查找**。

#### ⭐ 什么是深分页问题？️

`limit m n` 工作原理就是先读取前面 `m + n` 条记录，然后抛弃前 m 条，然后返回后面 n 条数据，所以 m 越大，偏移量越大，性能就越差。所以 `limit 10000000, 100` 要比 `limit 0, 100` 的性能差的多，因为要先读取 `10000100` 条数据，然后再抛弃前面的 `10000000` 条。这就是**深分页**问题。

解决方法：

**使用主键/唯一键优化（推荐）**

**携带前一页的 ID 进行查询**。记住上一页的最大 ID，下一页查询的时候使用 `id > max_id_in_last_page` 进行查询。

```
-- 原始低效查询
SELECT * FROM table ORDER BY id LIMIT 100000, 10;
-- 优化后查询
SELECT * FROM table WHERE id > 100000 ORDER BY id LIMIT 10;
```

**业务层优化**

- **使用 "加载更多" 代替传统分页**
- **使用缓存存储热门页数据**

**其他方案**

- **使用 Elasticsearch 等搜索引擎处理分页**
- **考虑分区表或分库分表**

#### MySQL 单表一千万条数据怎么做分页查询？

这其实是一个大数据量查询的问题，一千万级别数据量的表的话，MySQL 还是可以扛得住的，即使是单表也问题不大，但是也需要**注意查询的性能以及深分页的问题**。这个问题主要是需要从以下几个方面来做思考和优化。

**索引使用**

一定要走索引查询。在索引上，尽可能的选择唯一索引（包括主键索引），因为唯一性索引的话查询起来会更快一点。如果能用到索引覆盖，尽可能通过索引覆盖来查询，这样就可以避免回表带来的消耗了。

**避免深分页**

想要做分页就需要**考虑深分页**的问题。而针对 1000 万的数据，当**查询后面的部分页的时候，就会出现深分页的问题**，所以需要想办法避免。比如通过**记录上一页的最大 ID 的方式来实现查询**：

```
-- 第 1 页
SELECT * FROM large_table WHERE id > 0 ORDER BY id ASC LIMIT 100;

-- 第 N 页（假设上一页最后一条数据的 id 为 500）
SELECT * FROM large_table WHERE id > 500 ORDER BY id ASC LIMIT 100;
```

还可以通过子查询来实现分页：

```
-- 定位第 N 页起点
SELECT *
FROM large_table
WHERE id >= (
    SELECT id FROM large_table ORDER BY id ASC LIMIT 1000, 1
)
ORDER BY id ASC
LIMIT 100;
```

这样可以避免因 `OFFSET` 导致的大量无效扫描。但是这个方案不适合 ID 不连续的场景。

以上都是常用的一些**解决深分页**的方案。

还有就是当需要查询后几页的数据时，**倒序查询**可以减少扫描数据量：

```
-- 查询最后一页
SELECT *
FROM large_table
ORDER BY id DESC
LIMIT 100;
```

#### 分库分表的场景如何进行分页查询？

从理论上来说，分页查询要在全局有序的情况下进行。但是在**分库分表之后，要想做到 "全局有序" 就非常难了**。**分库分表下的全局分页查询思路**如下：

- **普通全局分页查询**。分库分表情况下，需要查询的数据可能分布在不同的表中，分库分表中间件一般采用的都是**全局排序法**。假如要查询的是 `**LIMIT x OFFSET y**`。那么分库分表中间件会把查询改写为 `LIMIT x+y OFFSET 0`，然后**把查询请求发送给所有的目标表。在拿到所有的返回值之后，在内存中排序，并且根据排序结果找出全局符合条件的目标数据**。这种方案性能很差。在内存中可以使用归并排序算法来给返回的结果排序。
- **禁用跨页查询**。禁用跨页查询，意思就是要求用户只能从第 0 页开始，逐页往后翻，不允许跨页。也就是在每一次查询条件里面加上上一次查询的极值，也就是最大值或者最小值。
- **平均分页**。把查询平均分配到各个表。在一些**对顺序、精度要求不严格的场景下是可以用的**。例如浏览页面，只需要返回足够多的数据行，但是这些数据具体来自哪些表，用户可能并不关心。
- **换用其他中间件**。比如使用 NoSQL 数据库，使用 ES 等。
- **引入中间表**。**额外存储一份数据，只用来排序**。这个方案里面就是在中间表里加上排序相关的列。为了保障中间表数据一致性，**可以使用 canal 监听 binlog，进行异步更新这个中间表**。

### JOIN 联结查询优化

少用 JOIN，JOIN 操作一般可以放到**业务层**进行。

#### JOIN 查询有啥算法？

主要是两个算法：

- 嵌套循环连接(NLJ)算法。
- 基于块的嵌套循环连接(BNL)算法。

#### 不使用 JOIN 如何做关联查询

如果不能通过数据库做关联查询，那么需要查询多表的数据的时候要怎么做呢？类似的问题：**跨不同数据库的 JOIN 如何实现**？

常见做法：

1. **在内存中自己做关联**。即先从数据库中把数据查出来之后，在代码中再进行二次查询，然后再进行关联。
2. **数据冗余**。把一些重要的数据在表中做冗余，这样就可以避免关联查询了。
3. **构造宽表**。就是基于一定的 JOIN 关系，把数据库中多张表的数据打平做一张大宽表。
4. **使用搜索引擎**。

#### 什么是驱动表？如何选择驱动表？

驱动表是表连接中的基础表，也就是通过**驱动表的数据结果集作为循环基础数据，然后一条一条的通过这个结果集的数据作为过滤条件到被驱动表中查询数据**，然后再做合并。**驱动表在 SQL 语句执行的过程中先读取，被驱动表在 SQL 语句执行的过程中后读取。**驱动表的选择会决定着一条 SQL 的执行效率。所以一条 SQL 中，该使用哪张表作为驱动表，其实是优化器决定的。

优化器选择驱动表的原则是：更好的访问性能和筛选性能，一般会做以下几个方面的考量：

- 表大小：**小表作为驱动表可以更快地被扫描和匹配，所以优化器倾向于选择较小的表作为驱动表**。
- 索引：可以利用索引进行加速访问的表作为驱动表可以提升效率。
- where 条件：如果查询中包含过滤条件，优化器会选择能够使用过滤条件进行筛选的表作为驱动表，以减少后续的匹配操作。

#### MySQL 为什么是小表驱动大表，为什么能提高查询性能？

进行联接查询时，**MySQL 通常会选择较小的表作为驱动表**，然后在较大的表中查找匹配的记录。这种策略的**核心思想是减少扫描和比较的次数，从而提高查询性能**。

假设有两个表：`employees`（1000 条记录）和 `departments`（10 条记录），并且要进行以下查询：

```
SELECT e.name, d.department_name
FROM employees e
JOIN departments d ON e.department_id = d.id
```

在不考虑 `hash join` 等其他链接方式，只考虑 `nested loop joi`n 的情况下，其实**执行的次数是笛卡尔积**，即：

```
for(1000){
    for(10)
}

和

for(10){
    for(1000)
}
```

但是假设 `employees.department_id` 和 `departments.id` 都有索引的情况下就不一样了，因为索引的查询是比较快的，其复杂度是 `log(n)`。那么：

- 小表驱动大表，复杂度为：`O(10) * O(log 1000)`
- 大表驱动小表，复杂度为：`O(1000) * O(log 10)`

这样一算的话，就非常清楚了，肯定是**小表驱动大表的整体的复杂度更低**！

### COUNT 查询优化

#### count(1)、count(\*) 与 count(列名) 的区别

`**COUNT(1)**` **和** `**COUNT(*)**`**表示的是直接查询符合条件的数据库表的行数。而**`**COUNT(列名)**`**表示的是查询符合条件的列的值不为 NULL 的行数。**

除了查询得到结果集有区别之外，在性能方面`COUNT(*)`是约等于`COUNT(1)`的，但是`COUNT(*)`是 SQL92 定义的标准统计行数的语法，因为是标准语法，所以 MySQL 数据库进行过一些优化。

#### ⭐️ 为什么 COUNT 全表查询这么慢？

**InnoDB 存储引擎的表 MySQL 不会维护表的总记录行数，因为需要实现 MVCC，数据可能有多个版本，所以并不好统计总数，查询 COUNT 需要实时计算**。所以类似的语句在 InnoDB 引擎上执行起来特别慢。

#### COUNT(\*) 非常慢该怎么办？如何快速查询全部数据的总数量？

优化 COUNT 的思路也比较简单，整体来说就是**不要去表里面 COUNT**。

- 如果业务要求精度不是很高，可以考虑**把总数维护到 Redis 里**。插入或删除表数据行的时候同时维护 Redis 里的表总行数 key 的计数值(用 INCR 或 DECR 命令)。但这可能存在数据不一致。
- **使用一个额外的表来记录总数**。

如果用了 Redis 来维持总数，那么就会涉及**数据一致性**的问题。也就是如果插入数据库成功，但是更新 Redis 上的总数失败了，应该怎么办？

1. 如果数据短时间不一致但是业务可以接受的话，那么就可以考虑**异步刷新 Redis 上的总数**。
2. **使用 Canal 之类的工具监听 MySQL binlog，然后刷新 Redis 上的总数**。

## 服务器与基础设施优化

### MySQL 参数配置

可以通过设置一些参数控制 MySQL 的行为，对数据库进行优化。

#### MySQL 优化最重要的几个参数配置有哪些？

最重要的几个参数：

- **innodb_buffer_pool_size**。配置 **InnoDB 缓存池大大小**，一般为物理内存的 `60%-70%`。
- **innodb_log_file_size**。配置 **redo log 空间的大小**，默认值为 `48M`。

#### ⭐️ 常见的配置参数有哪些？

**缓存池大小相关的配置**

如果大部分都是 InnoDB 表，那么 InnoDB 缓冲池(Buffer Pool)或许比其他任何东西更需要内存。InnoDB 缓存池会缓存索引、缓存行的数据、自适应哈希索引、插入缓冲(Insert Buffer)、锁以及其他内部数据结构。总之 **InnoDB 严重依赖缓冲池，必须确认为它分配足够的内存，增加这个值可以减少磁盘 IO**。

**参数** `**innodb_buffer_pool_size=40G**` **配置 InnoDB 存储引擎 buffer pool 缓存池大小，一般为物理内存的** `**60%-70%**`。但是也要小心这个参数过大，物理内存不足，容易触发操作系统 swap。

**日志与 IO 相关的配置参数**

一些重要参数：

- **参数** `**innodb_log_file_size**`。配置 redo log 空间的大小，默认值为 `48 M`。
- **参数** `**innodb_log_buffer_size**`。设置 **redo log buffer** 大小，默认 `16M`，最大值是 `4096M`，最小值为 1M`。`YFD 设置为 `8388608(8M)`。
- **参数** `**innodb_flush_log_at_trx_commit**`。配置 redo log 的从**内存缓冲区写入磁盘的策略**(日志缓冲刷新的频繁程度)。

**数据库连接相关的参数**

- **参数** `**max_connections**`。配置最大连接数，不能过大，因为每个连接都会占用不少内存。
- **参数** `**wait_timeout**`。配置连接超时时间，默认 8 小时。

**并发相关的配置参数**

- 参数 `**innodb_thread_concurrency**`。限制**一次性可以有多少线程进入内核，0 表示不限制**。
