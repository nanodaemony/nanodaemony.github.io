## 基本操作

### 基本工作流程

日常开发中最常用的命令序列是 `add`-> `commit`-> `push`。

```
# 检查当前工作区和暂存区的文件状态（非常重要）
git status
# 将文件的修改添加到暂存区（Index）
git add <文件名>        # 添加特定文件
git add .              # 添加当前目录所有【新文件】和【修改过的文件】
git add -A             # 添加所有【修改】和【删除】的文件变更[1](@ref)
# 将暂存区的内容提交到本地仓库，并附上说明
git commit -m "清晰的提交说明信息"
# 一个快捷操作：自动添加所有已跟踪文件的修改并提交（对新文件无效）
git commit -am "清晰的提交说明信息"
```

### 分支管理

分支是 Git 的核心功能，能让你在不同功能或修复上独立工作。

```
# 查看所有分支
git branch              # 查看本地分支
git branch -a           # 查看所有分支（包括远程）[1](@ref)
# 创建与切换分支
git branch <新分支名>    # 基于当前提交创建新分支
git checkout -b <新分支名> # 创建并立即切换到新分支[1](@ref)
# 切换到一个已存在的分支
git checkout <分支名>
# 合并分支：将指定分支的修改合并到当前分支
git merge <要合并过来的分支名>
# 删除分支
git branch -d <分支名>   # 安全删除（仅当分支已合并时）
git branch -D <分支名>   # 强制删除（即使未合并）[1](@ref)
```

### 远程仓库协作

与团队成员协作或备份代码需要操作远程仓库。

```
# 查看已配置的远程仓库
git remote -v
# 添加一个新的远程仓库别名
git remote add <别名> <远程仓库URL>
# 从远程仓库获取更新（不会自动合并到工作区）
git fetch <远程仓库别名>
# 拉取远程分支最新更改并【合并】到当前本地分支
git pull <远程仓库别名> <远程分支名>
# 推荐使用 rebase 方式拉取，使历史更线性
git pull --rebase <远程仓库别名> <远程分支名>
# 将本地分支的提交推送到远程仓库
git push <远程仓库别名> <本地分支名>
# 首次推送本地新分支到远程，并建立跟踪关系
git push --set-upstream origin <分支名>[1](@ref)
# 删除远程分支
git push <远程仓库别名> --delete <远程分支名>
```

### 查看与比较历史

了解项目的演变和对比差异至关重要。

```
# 查看当前分支的提交历史
git log
# 以更简洁的单行格式查看提交历史
git log --oneline
# 查看所有操作的历史记录（包括被“丢弃”的commit）
git reflog
# 比较工作区与暂存区的差异
git diff
# 比较暂存区与最新提交的差异
git diff --staged
# 比较两个具体提交之间的差异
git diff <提交ID1> <提交ID2>
```

### 撤销与恢复操作

这些命令很强大，但请谨慎使用。

```
# 丢弃工作区中某个文件的修改（危险操作）
git checkout -- <文件名>
# 将文件从暂存区撤回到工作区（取消 add）
git reset <文件名>
# 撤销提交（可软可硬）
git reset --soft HEAD^   # 撤销commit，但保留修改在暂存区
git reset --hard HEAD^   # 彻底丢弃最近的提交和所有修改【谨慎使用】[4](@ref)
# 创建一个临时储藏点，保存当前工作区和暂存区的修改
git stash
# 恢复最近储藏的修改并删除储藏记录
git stash pop
```

### 标签管理

**标签常用于标记发布节点**（如版本号）。

```
# 列出所有标签
git tag
# 创建一个新标签（指向当前提交）
git tag <标签名>
# 创建一个带注释的标签
git tag -a <标签名> -m "标签说明信息"
# 将指定标签推送到远程仓库
git push <远程仓库别名> <标签名>
# 将所有本地标签推送到远程仓库
git push <远程仓库别名> --tags
```

### Cherry Pick

Git 的 `cherry-pick`是一个强大且精准的命令，**它允许你将一个或多个特定的 commit 从任何一个分支 “采摘” 出来，然后应用到当前分支。这就像从一堆樱桃中\*\***只挑选出最成熟、最需要的那几颗\*\*，而不是把整串都摘下来。

**何时选择 Cherry-Pick？**

|                   |                                                                                        |                                        |
| ----------------- | -------------------------------------------------------------------------------------- | -------------------------------------- |
| **操作**          | **适用场景**                                                                           | **特点 (Note)**                        |
| `**cherry-pick**` | **精准地移植单个或少量特定提交**，例如热修复、从废弃分支抢救代码、应用不相关的功能点。 | 产生新提交，历史记录可能不线性。       |
| `**merge**`       | **整合完整的特性分支**，保留完整的分支历史和上下文。                                   | **保留分支结构，会产生一个合并提交**。 |
| `**rebase**`      | **在推送前整理本地提交历史**，使其更清晰、线性。常用于个人特性分支。                   | **重写历史，使提交历史更整洁。**       |

- **批量采摘提交**：可以一次性应用多个提交，只需将它们的哈希值依次列出，用空格隔开：

```
git cherry-pick <commit-hash1> <commit-hash2> <commit-hash3>
```

- 也可以使用**范围语法**来应用一串连续的提交（注意区间表示法）：

```
git cherry-pick <start-commit-hash>^..<end-commit-hash>  # 包含 start-commit
```

- **只应用变更，暂不提交**：使用 `-n`（或 `--no-commit`）选项。命令会应用修改，但不会自动创建新的提交，而是将更改保留在工作区和暂存区。这允许你审查修改，或**将多个提交的变更合并成一次提交**。

```
git cherry-pick -n <commit-hash>  # 应用更改，但不要提交
```

- **保留原提交作者信息**：使用 `-x`选项。这会在新提交信息中追加一行 `(cherry picked from commit ...)`，便于追踪来源。这在团队协作中很有用。

```
git cherry-pick -x <commit-hash>
```

- **遇到冲突时选择 “他们的” 或 “我们的” 版本**：在冲突时，可以用 `-X ours`（选择当前分支的更改）或 `-X theirs`（选择要采摘过来的提交的更改）来指定偏好策略，但这需谨慎使用，以免意外覆盖代码。

```
git cherry-pick -X theirs <commit-hash>
```

## 实用技巧

### 修改最后一次提交信息

如果刚刚提交但写错了信息，可以使用：

```
git commit --amend -m "新的提交信息"
```
