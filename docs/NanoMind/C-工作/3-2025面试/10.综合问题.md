#### ⭐ 做了哪些限流降级的事情？

> **同步练目录接口降级**

同步练目录接口是学练机的**首页开屏接口**，业务重要性非常高，流量也最高，且出现过几次线上故障，影响很大。

**降级主要是保障用户的核心的练习目录功能可用**。可降级的旁路逻辑：

- **最近练习的章节**：基于 Redis 做的。
- **各个章节练习成绩**：基于练习记录查询做的。
- **人教社的点读**：会在目录上**外显点读的信息**，需要调人教社的接口。

完整方案：

- 新增一个**一键降级开关**。出问题时可以及时关闭旁路逻辑。
- 对首页目录接口的响应值 + 章节出题信息**进行缓存**。遇到**上游依赖的内容服务出故障时**，直接使用缓存的数据。这个是做到**基本可用**。缓存过期时间为 24 小时。
- 进行**故障演练**，模拟上游接口全部挂掉的场景。

> **服务端题目渲染降级**

采用服务端对题目进行渲染时，**如果渲染失败，最终会降级到使用客户端自己渲染**。

> **拍照检查 OCR 模型自动降级**

优先使用公司自研的模型进行 OCR，如果失败次数过多，则自动降级为使用更贵、响应更慢的 Google OCR。这里如果失败会进行**自动熔断降级**。

### 服务与资源隔离拆分

#### ⭐ 为什么做资源隔离？

因为前期业务快速迭代，各个业务都放到了同一个服务，数据库也是复用的同一个集群，前期流量小的时候还顶得住。

一次故障中，原因是整个数据库集群流量非常高，且有很多慢查，加上流量高峰期的时候缓存过期，同时遇到代码里面有循环查库，导致依赖的上游内容组的数据库集群压力增大，超时变多，最终雪崩导致整个集群都挂了。不只是墨水屏，其他的口算、搜题业务也因此受影响了。

因此**后面就开始全面考虑做资源隔离了**。

#### ⭐ 做了哪些资源隔离与拆分工作？️

这里主要是几个方向，**微服务拆分、核心服务独立部署隔离、数据库拆分隔离**等。

> **微服务拆分**

因为前期业务快速迭代，各个业务都放到了同一个项目中，这个其实已经逐渐变成一个**大单体**了。服务启动缓慢、依赖众多，存在牵一发而动全身的问题。因此考虑把单独的业务模块拆分迁移到新的微服务中，这块差分也是**根据领域模型**进行拆分的。

> **核心服务独立部署隔离**

**使用相同的项目，部署多个不同的服务**。比如口算这边将拍照检查服务的快、慢接口进行隔离部署；墨水屏这边针对最核心的同步练相关的业务也**单独部署了一个服务**，防止其他的业务场景故障而导致核心业务受影响。因为遇到过某个非核心业务阻塞导致吃完线程，导致所有请求阻塞的问题。

> **数据库拆分集群隔离**

**早期的很多数据库都放到一个集群的**，后面数据库多了且数据量越来越大，导致集群性能容易出问题。因此按照业务进行了数据库集群的拆分，同时预估了各个集群未来 3 年的可能的数据量级。

#### ⭐ 隔离有哪些考虑的方向？️

**业务与数据的关联性**，比较独立的块进行隔离。

业务与数据的**重要性和流量**，越重要且流量高的单独隔离。

**平衡成本，隔离往往导致成本更高，也不是无脑都隔离了**。

### 慢查治理优化

#### ⭐ 做了哪些慢查 SQL 优化工作？

> **识别与监控慢 SQL**

- 通过慢 SQL 日志识别慢 SQL。**建立慢查告警机制，超过 100ms 的就认为是慢查**。
- 通过 **EXPLAIN 分析执行计划**，重点关注 `type、key、rows、Extra` 等字段。
- 使用性能监控工具如 `Prometheus + Grafana` 监控数据库性能指标。

> **索引相关**

- **索引失效排查**：新的业务查询并没有主要到新的字段没有索引，因此导致慢查，解决是加索引。也可能因为格式转换问题导致索引失效。
- **索引合并**：有的冗余联合索引进行了合并与删除。业务发展导致不同的人建立了重复的联合索引。
- **修复 SQL 注入问题**。
- **减少回表操作**：使用 SELECT 字段替代全查等。

> **深分页相关**

- **处理深分页问题\*\***。后台读取数据时，如果指定读取后面的页数就会出现深分页问题\*\*。解决方式是采用 cursor 方式进行数据读取。

> **其他**

- **对大表进行优化。分库分表 + 数据归档**。
- **使用 ES 替代 MySQL**。

### 大表优化与分库分表数据迁移

#### ⭐ 做过什么大表优化工作？️

> **拍批拍搜相关大表优化**

背景：拍批拍搜相关的表记录了用户拍照检查、作业批改等的所有数据，包含业务数据 + 研究组返回的批改结果，数据量巨大。前期对于每个场景仅设计了一个单表，**后面单表数据量超 6 亿，单表占用磁盘空间超 1T**。核心原因在与**表里面会把批改原始结果存下来，因此数据量巨大**。

优化方案：

- **将批改原始结果的大字段拆开**，将其存储到 OSS 中，表中只存文件 ID。有的地方也把大数据存到了 HBase 中。
- **进行分库分表**。根据预估的数据增量，拆分为了 8 库 256 张表。
- **定期进行数据归档**。这个业务的数据仅给用户展示 1 年内的记录，因此可以定期归档一年前的数据，节省磁盘空间。

> **练习记录的分库分表**

练习记录是练习域的聚合根，非常重要。**前期也只是设计了单表**。其处理方式与上面的一致。

#### 一定需要分库分表吗?

不一定，看**业务场景**。有的不是很重要但是数据量很大的场景，没必要用 MySQL 做分库分表，因为引入分库分表会更加的复杂，也会引入一些新的问题。这种场景下**可以采用定期归档的方式优化单表性能**。也可以选用其它的存储实现。

#### 分库分表怎么做的？

> **练习记录分库分表**

练习记录表进行分库分表，共创建 16 库 512 表。**分库分表键为 userId**，切分策略为 `userId % 512`。

> **题目渲染结果的分库分表**

对题目渲染结果进行存储，分表键为题目 ID。

#### 分库分表是否需要使用一致性哈希？

分库分表可以结合一致性哈希算法，但**并不是必须的**。是否使用一致性哈希取决于具体需求和场景。

比如分库分表**规模固定且不计划频繁扩容、使用简单的哈希取模或范围分片已能满足需求时**，就不需要使用一致性哈希。**正常来说一般是会在前期进行容量评估的，所以容量不够的情况下，再次扩容一般也是个低频的操作，没必要用一致性哈希**。但是使用一致性哈希有一个好处就是在确实需要迁移的情况下，能够减少数据迁移量。

#### ⭐️ 分库分表后遇到哪些问题？

> **旧历史数据无法定位的问题**

前期的练习记录 ID 是通过雪花 ID 做的，分库分表后历史数据被分配到了不同的表中，导致原来有些只根据练习记录 ID 查询数据的场景无法定位到数据的位置了。所以后面对于历史数据建立了一张新的映射表，查询的时候先查询到表的位置，再进行二次查找。

> **需要再次扩容的问题**

前期容量评估不足，导致后面需要重新扩容。这个就比较麻烦了。

#### ⭐⭐ 数据迁移怎么做的？️

> **练习记录数据表的分库分表数据迁移**

比如 YFD 的核心业务是练习记录。这个表在开始设计时是单表的，后面由于单表数据量过大(6 亿)，导致出现一些**慢查询**，所以决定对其进行分库分表，这里就涉及数据迁移。

具体流程如下：

![](/zimg/10.%E7%BB%BC%E5%90%88%E9%97%AE%E9%A2%98-1775395844642.png)

以上就是一个数据迁移的完整流程，**从最开始的读写都从旧表进行，最终变成读写都从新表进行**。具体流程如下：

- 写旧库读旧库。最开始的状态。
- **双写读旧库：增量数据迁移**。**修改代码进行双写**，优点是好控制，比如异常回滚只需要一个动态配置即可，缺点是侵入性比较严重。另一种方法是**采用 Canal 进行增量同步**，优点是解耦。我们采用的代码双写方式，原因是**更好控制流程且更加的灵活**，比如我们可以提前使用**白名单机制进行数据验证与测试**。
- **增量数据校对**。**因为增量数据需要双写，双写是有可能失败的，所以需要有一定的机制可以发现这些双写失败的数据**。我们采用的方案就是做旁路验证：即在业务完成后，**将当前读旧库返回的数据缓存下来，再通过一个异步线程走相同的逻辑并读新库，最后对比一下两者的区别**，如果有问题则进行报警。同时还可以进行 JOB 的比对，比如从某个 ID 开始读取数据，一条一条的进行对比。如果有不一致的数据，就**以旧库为主进行校正**。
- **增量数据的更新**。在数据迁移过程中，业务数据是会发生不断地变化的，前面的双写只把新发生的 insert 当做增量。那这个过程中，如果发生了 `update` 就需要更新增量数据。这里需要做一个逻辑，那就是在 `update/delete` 的时候，**判断一下这个数据是增量的还是存量的**，其实就是判断新表中是否有这个数据，**如果有就需要做双写更新，如果没有，就说明这是个存量数据，只更新旧表就行了**。不用担心他们会丢，因为后续会做存量同步，这部分更新也会带过来的。当然这里如果使用 Canal 就会非常方便了！
- **存量数据迁移**。做好了增量数据迁移和核对之后，就可以**把存量的数据也迁移过来**了。这一步**主要是 DBA 来操作**。做好断点续传即可。
- **存量数据校对**。做数据核对，把新表和旧表的数据做逐条核对，确保数据是一模一样的。
- **双写读新库(先切读流量再切写流量，同时进行切流的灰度放量)**。在确保数据都完成迁移之后就可以做**切流**了，但**一般是先把读请求切流到新的表上，跑一段时间，确保存量和增量数据都没问题，再切流写请求**。在切流的时候，需要有灰度的机制，基于 userId 或者白名单进行灰度。这一步在切流之后，旁路的策略也需要跟着调整。因为切流后读新表了，那么旁路就需要切换到读旧表。
- **增量数据校对**。**再次做一次数据校对。此时有问题还能回滚**。
- **切流写新库**。不能回滚了。最后一步要把双写去掉，完全变成**单写**了，这一步之后，旧表的数据就不会再做更新了。所以在这一步之前，要做好足够的验证和核对，确保数据无误之后，在做谨慎操作。

这里由于涉及机房和地域的迁移，使用双写的方式并不契合，所以**只能采取停机迁移的方式**。

#### ⭐ 线程池的参数是如何确定的？️

> **批量提交批改的场景**

需要调研究组接口进行批改，**研究组\*\***接口的响应时间平均是\*\* **600ms\*\***，单机上请求\***\*第三方的 QPS 约为 200 每秒**。机器是 8C 16G 的。

L (并发请求数) = 200 QPS × 0.6 秒 = **120**。因此**理论上需要 120 个线程**才能完全处理 200QPS 的请求（无队列等待）。

由于机器是 8 核的，且任务是 IO 密集型的，**线程大部分时间是在等待响应，所以可以适当的配置线程数是 CPU 的** **10-20 倍**。大概就是 80-160，所以**核心线程数配 80 就可以**了。经验表明，IO 密集型任务中，线程数超过 200 后性能可能下降，线程数超过一定阈值后，**上下文切换开销**会显著增加（尤其对 8 核 CPU）。

**最大线程数用于应对突发流量或响应时间波动，\*\***通常为核心线程数的 1.5-2 倍\*\*，注意不要超过 CPU 的承受上限（建议小于 150）。

因此线程池参数：

- **核心线程数：80**。
- **最大线程数：120**。
- 阻塞队列。**选用有界的队列防止 OOM**。可以稍微大一些。取 500。
- **空闲线程存活时间**。取 30s。
- **拒绝策略：丢弃任务，快速失败**。

注意：**响应时间波动时（如第三方接口变慢或者有长尾请求），线程长时间阻塞**，因此需要合理设置超时时间。

```java
// 参数定义
int corePoolSize = 80;
int maxPoolSize = 120;
int queueCapacity = 100;
long keepAliveTime = 30;
TimeUnit unit = TimeUnit.SECONDS;

// 创建线程池
ThreadPoolExecutor executor = new ThreadPoolExecutor(
    corePoolSize,
    maxPoolSize,
    keepAliveTime,
    unit,
    new LinkedBlockingQueue<>(queueCapacity),
    Executors.defaultThreadFactory(), // 可选自定义ThreadFactory
    new ThreadPoolExecutor.CallerRunsPolicy() {
        @Override
        public void rejectedExecution(Runnable r, ThreadPoolExecutor e) {
            monitor.rejectedTaskCount.increment();
            super.rejectedExecution(r, e); // 降级处理
        }
    }
);

// 建议预热核心线程
executor.prestartAllCoreThreads();
```

执行任务时**可以用 Future 设置超时时间**。

```java
ExecutorService executor = Executors.newFixedThreadPool(10);

Future<String> future = executor.submit(() -> {
    // 调用第三方接口（可能耗时）
    return callThirdPartyAPI();

try {

    System.out.println("Result: " + result);

    // 超时处理
    future.cancel(true); // 中断任务线程
    System.out.println("Task timed out");
} catch (Exception e) {
    // 其他异常处理
}
```

#### 数据库链接数有多少？

使用连接池能有效的提升性能，减少数据库连接建立的开销，但是连接池的大小并不是越大越好，**每申请一个长连接都会在物理网络上建立一个用于长连接维护的线程，而线程的执行跟物理机的 CPU 核数有关**。理论上一个 N 核的服务器将连接池设置为 N 最佳，每一个核同时处理一个线程，超过 N 的并发就有线程上下文切换的开销。除此之外需要考虑的是当前服务所在机器的性能，网络状况，数据库机器性能，数据库特性等等。

本文会**针对 commons-db 使用的 dbcp2 数据库连接池进行参数设置的说明**。

- maxTotal：**可以在这个池中同一时刻被分配的有效连接数的最大值。默认 1000，建议在 2.5K 以下，2.5K 是压测的经验值**。数据库实例（包括实例下所有库）最大连接数，即实例级别最大连接数建议在 8K 以内。
- maxIdle：连接池中可以保持空闲状态的最大连接数，**超出设置值之外的空闲连接在归还到连接池时将被释放**。设置得太小会导致在高并发且流量存在潮汐现象的情况下频繁销毁以及创建连接，设置得太大会导致流量高峰过去之后大量连接占用不释放，影响数据库性能和应用内存占用。默认值 100。

#### ⭐ 是否处理过死锁问题？️

> **用户多次点击引发死锁**

有一个业务是**用户提交自己的作答记录，会去更新题目的状态**。SQL 类似于：

```
UPDATE mission_process SET question_status = 1 WHERE id IN (1, 2, 3);
```

由于**用户客户端**的问题，**用户可能同时快速点击两次**，导致引发了死锁**。数据库的隔离级别是**可重复读\*\*。

**原因**：当两个事务同时执行这个语句时，MySQL 在获取行锁时可能不是按照固定的顺序（如 ID 顺序）来获取的，事务 A 可能先锁 ID1，再锁 ID2；而事务 B 可能先锁 ID2，再锁 ID1。这就导致了死锁。注意 MySQL 不会一开始就同时锁定所有三个 ID，而是**逐行查找并锁定满足条件的记录**。所以可能执行更新的时候，A 持有 ID1 的锁，B 持有 ID2 的锁，造成循环等待就会死锁了。

解决方法就是添加 ORDER BY 确保锁**按固定顺序获取锁**：

```sql
UPDATE mission_process SET question_status = 1  WHERE id IN (1, 2, 3) ORDER BY id;
```

#### ⭐️ 消息堆积了怎么办？

首先需要明确的是，MQ 堆积是正常的，因为 MQ 有个重要的作用就是削峰填谷，既然能起到削峰填谷的作用，那就意味着它需要去接收更多的消息，然后放到自己的队列里面，下游再慢慢消费。所以出现堆积的情况不要慌，也不一定要立刻就进去解决。

MQ 堆积的这个问题一旦发生了，比如线上有告警了，处理过程应该是：

1. 先去**定位具体什么场景**，哪个 topic 的消息堆积了。
2. 看下**当前的堆积情况是否严重**，是否在减缓。
3. 查看**上游流量情况，是否有营销活动，或者定时任务在运行**。
4. 分析下**堆积导致的延迟是否可以接受**。
5. 考虑**扩容增加消费者提升消费速度**。
6. 优化代码，进一步解决堆积的问题。

解决一般是两种方案：

- **临时方案**：考虑扩容，增加消费者数量，当然这必须要考虑下游是否能够承受得了。实在不行，可以评估下是否可以丢弃消息，后面再重置消费点位。
- **长期方案**：分析是哪些地方导致了消息堆积。比如消费者使用的是一条一条拉数据而不是批量拉取、消费者性能有问题等。也可以考虑堆积时做一些降级操作：比如对生产者限流；或者在消费者组将消息保存起来，后面再慢慢消费。

#### ⭐️ Kafka 与 RocketMQ 的区别（问的巨多）

1. **定位不同**：

- Kafka：**高吞吐流处理**（日志、实时计算）。
- RocketMQ：**金融级消息队列**（事务、顺序消息）。

2. **消息模型**：

- Kafka：分区顺序，**不支持回溯**，流式消费。
- RocketMQ：队列模型，**支持事务/回溯/延迟消息**。

3. **吞吐与延迟**：

- Kafka：**吞吐更高**（百万级），但消费延迟依赖分区。
- RocketMQ：吞吐适中（十万级），**端到端延迟更稳定**。

4. **可靠性**：**RocketMQ 支持 kafka 所不具备的“同步刷盘”功能**，在单机可靠性上比 kafka 更高，不会因为操作系统 Crash 而导致数据丢失。

- Kafka：ISR 副本同步，可能丢消息（需配置 `acks=all`）。
- RocketMQ：**强一致性**（同步刷盘），内置事务。

5. **适用场景**：

- 选 Kafka：日志、流计算（Flink/Spark）。
- 选 RocketMQ：订单、支付等**强一致性业务**。

**一句话总结**：

**Kafka 适合流式大数据，RocketMQ 适合事务性业务**。
