### 综合问题

#### 故障处理流程与规范是怎么样的？

> **周知故障相关人员**

故障发生时应该首先周知故障相关方大致影响范围，并给出预计恢复时间。

- **研发团队**：主要责任人开始故障处理流程，其他研发同学待命关注进展，尽量暂停不必要的上线操作，避免出现更多的问题，增加处理难度。
- **产品/运营**：如果不能排除 B 端影响，应该优先考虑暂停 B 端操作。
- **测试团队**：测试团队可协助进行故障范围的确定。
- **客服团队**：应对可能出现的客诉。

通知渠道：主要是企业微信群。

> **建立故障处理群**

为了故障处理的高效和信息及时同步，应该**建立单独的故障处理群，邀请故障处理相关方入群**。主要包括如下人员：

- 研发主管：主要作为故障指挥，指导相关人员进行故障处理。
- 服务端研发团队：主要责任人开始进行故障原因诊断。
- OPS：可以协助排查问题。
- DBA：可以协助排查问题。

如果是非工作时间或者因其他原因无法进行及时沟通，尽量使用线上会议来及时同步信息，避免只依赖于群聊。故障处理过程中应该**尽量将相关分析及操作记录发送至故障处理群中，一是为了同步信息至各方，二是为了方便故障过后进行故障复盘**。

> **故障原因快速诊断**

故障原因诊断可以优先从以下几方面考虑入手：

- **看看是否有变更**。确认最近一段时间相关服务 是否部署过新代码、是否修改过配置、是否执行过线上 SQL 等（可能上线之后不一定会立马暴露问题，而是由于别的外因触发导致故障发生，比如定时任务执行、高峰期流量大等）。
- 确认产品/运营是否修改过相关配置。需要注意各个配置平台。
- **确认 DBA/OPS 是否修改过相关配置**。例如机器扩容，流量调整，流量迁移等。
- **确认是否是由于前端上线或者客户端发版导致**的某些接口流量激增等引起的。

可以使用一些**工具/平台/监控**来配合排查问题。

> **故障处理修复**

\*\*恢复服务：故障修复应首先考虑恢复服务，对外正常提供服务。

- 如果是由于部署新代码、修改配置、执行 SQL 等操作引起的故障，优先考虑**回滚服务/配置/SQL**。如果快速定位到故障原因，修复成本较低（一般为小的代码 Bug），并且处于业务低峰期，也可以考虑修复之后上线。
- 如果故障时间比较长，可能造成上游服务发生雪崩效应，**应优先考虑采取降级策略**，牺牲一小部分服务以避免影响范围扩大。

#### ⭐️ 稳定性建设有哪些方法？如何提升系统稳定性？

稳定性不只是常见的熔断，降级这些。**从最开始的业务评估、找出核心业务、梳理核心指标等都是，稳定性都是基于这个来做的。到后面技术方案、研发规范、上线部署、Review 等规范，都是稳定性保障的一部分**。

可以分为多个稳定性保障的阶段：

1. **日常风险清理**。错误日志处理、慢查告警处理、工单反馈处理。每周对齐，跟进，排任务解决。
2. **日常值班**。每周固定人员值班，及时响应。
3. **监控告警**。梳理核心业务指标与监控指标，制定告警规则，通过监测指标来判断业务是否正常。
4. **数据库治理**。慢查优化、大表优化、表空间优化、数据隔离。
5. **微服务治理**。服务拆分、核心链路梳理、限流降级、缓存、熔断、备份。
6. **上线规范制定**。上线时间约束、CheckList 周知、上线周知。
7. **故障处理预案**。制定完整的故障处理流程，针对核心场景制定故障处理 SOP 手册。

#### ⭐ 缓存过期+内容侧慢查导致服务异常 ️

现象：墨水屏、口算、搜题等多个业务线核心功能不可用，许多接口超时，内容侧服务基本不可用。

直接原因：

- **内容 MySQL 数据库集群未做业务线隔离，日常负载重**。该 MySQL 集群日常慢查询 SQL 非常多，整体流量高，当时慢查询 SQL 增多，出现**故障的夜间高峰期达到性能拐点**。且不同团队业务依赖的数据库都在这个集群上，运营生产后台、给 C 端提供服务的实时数据读取接口等 SQL 查询都运行在这个集群上，业务上没有做隔离。**集群中有数据库 22 个，存储空间 4.6T，单节点读 QPS：15K+**。
- C 端依赖 `leo-question` 服务的 `getQuestionCountByKeypointIdsForMegrezEx` 接口获取章节下每种类型的题目数量的**接口由于业务复杂，本身性能较差（日常响应就在 2s 左右）**。因此业务侧会对此 RPC 接口的返回值进行**本地缓存**，故障发生期间，**缓存的 <章节,题目数量> 数据全部失效（缓存过期时间为五分钟）**，因此在缓存失效期间（2s 左右）的请求流量就会全部打到 `leo-question` 服务进行数据库查询。而故障期间 DB 压力大，这个接口响应很慢，响应时间可能超过 5s，因此请求的窗口更大了，由此引入了更多的慢查询。同时，故障期间用户**可能出现失败重试**，导致该接口 QPS 进一步突增（在故障开始之前，QPS 只是常规的 5QPS，在 21:01 也为 10 QPS ，故障期间峰值达到了 100+，相比平时增加了 10 几倍），而 `leo-question` 服务查询 DB 的数据量大，产生更多慢查询，最终导致雪崩。在故障之前，C 端缓存命中率比较健康：在 20:59 时缓存命中率为 90%，21:00 为 86.6%。后面缓存命中率急剧降低。

#### ⭐ 压测中 parallel 并行流操作导致服务宕机 ️

现象：请求 `megrez-math` 服务的接口超时，导致客户端报错。**服务响应 504（网关超时）**。

原因：举一反三是一个非常吃研究组性能的场景，**所以为了摸清楚依赖的研究组服务能抗多少请求，就进行了一次压测**。举一反三是在**题目维度**进行的操作，每次可能有上百个题目同时进行举一反三，而业务代码中**使用了 stream 进行批量举一反三操作**，同时**使用了 parallel 进行并行操作**，这里面会调用研究组的接口，平均响应在 1s 左右，因此整体的操作耗时是非常长的。

**Stream 的 parallel 并行流，底层默认使用的是 fork join，并行任务的数量与 CPU 一致，当任务多了之后，就会使用原本的线程来执行任务了，也就是使用 Tomcat 的线程池来执行任务了，就可能导致 Tomcat 线程池被压满**。表现上就是所有的其它请求全部被 hang 住了，导致整个服务的其他接口都阻塞超时了（因此响应 504 了）。因此对于 IO 相关的任务，尽量不要用 Stream 的并行流。如果要用 Fork Join Pool，需要自定义来使用。

解决方式是改成了**自定义的线程池进行处理**。

#### 删除历史数据导致主从延迟巨大影响很多业务

现象：拍照检查的记录表单表数据 6 亿＋，所以准备**删掉三个月之前的数据**。执行删除过程中，导致整个**数据库集群的主从延迟增大(最高达到 200 秒+)**。由此影响了不少读集群从库的业务。

原因：数据库删除大量数据时候，**会产生大量的 binlog，这些 binlog 需要同步到从库进行应用，从而导致主从延迟增大**。不论是删除还是更新操作等，都是以**行为单位**记录到二进制日志（binlog）中的，然后再由从数据库来读取并执行这些日志。**当删除大量数据时，会生成大量的二进制日志，从数据库在同步应用这些日志时，会消耗相应的 I/O、CPU 资源，特别是当单个事务操作数据量非常大时，可能会造成从数据库同步延迟。大事务执行时间长, 容易造成主从延迟**。

> **如何改进？**

在**业务低峰期进行大量删除操作**。当需要**删除大量数据时，可以分批次删除，每次操作的数据量不要过大**，以避免产生太多的日志，防止主从同步延迟。比如每次删除 1000 行或者 10000 行，然后 sleep 一小会再继续删除。

> **如果是对这个数据量很大的表修改索引操作的话，会出现这种问题吗？**

**对大数据量的表进行索引修改，也可能会产生大量的日志，导致主从同步延迟**。比如使用 ALTER TABLE 命令添加或删除索引。ALTER TABLE 会**创建一个新的表，然后从旧表中复制数据，并重建索引**。这个过程会产生大量的写操作和日志，如果主库和从库之间的网络带宽、I/O 性能等有限，都可能导致主从同步延迟。如果修改索引的方式是通过 DROP INDEX 和 CREATE INDEX，虽然不需要复制数据，但**仍然会产生大量的日志**。DROP INDEX 会产生删除旧索引的日志，CREATE INDEX 会产生创建新索引的日志，这些日志在主库上产生，然后需要同步到从库上应用。因此如果数据量大，主从同步也可能出现延迟。

因此**修改大数据量的表的索引，同样建议在业务低峰期进行，尽量避免一次性产生大量的日志**。另外，也可以**考虑优化主从复制的配置**，提高主从复制的效率，降低主从同步延迟。

发生 OOM 时的**图片都比较大，读取过程需要较大的内存空间**。虽然图片请求产生的中间对象体积比较大，但是这些对象都是朝生夕死的，正常来说，就算堆满老年代，只要**触发一次 Full GC**，就会把无用的对象全部清理掉。但是**依然发生了 OOM**。由于发生 OOM 时的机器已经被替换，当时的 dump 文件也没法拿到，所以无法分析当时的具体情况。

但从 JVM 的配置可以发现问题。**服务使用的垃圾收集器为 G1**。

G1 收集器的特点：

- \*\*堆区分为 2048 个 region，当对象占用内存超过 region 的一半时将被视为大对象。大对象直接分配到老年代。
- JVM 参数 `-XX:G1MixedGCLiveThresholdPercent`。**此参数配置了只有存活对象低于该参数的 region 才可以在 MixGC 阶段被回收，默认值为 85%**。

这样可以大概定位到 OOM 的原因。线上机器 `-XMS=12G`，**计算 region 大小可以为：6M**。也就是超过 3M 的对象就会被直接分配到老年代，**所以当有大图片需要处理时，产生的对象很容易直接被分配到老年代**。同时**有的大对象体积超过了一个 region 的 85%，导致这些垃圾不会被 MixGC 回收**。

整体分析下：

- **大对象分配机制问题**：当对象超过 region 大小的 50%（这里为 3M）时，G1 会将其**直接分配到老年代**，图片处理产生的中间对象虽然 "朝生夕死"，但因为尺寸过大直接进入了老年代。
- **G1 的混合 GC 限制**：`G1MixedGCLiveThresholdPercent=85%` 意味着**只有老年代 region 中存活对象占比小于 85% 时才会被 MixGC 回收，如果有些图片对象确实大于了 85%，就暂时不会回收**。
- **Full GC 的局限性**：虽然 Full GC 会清理整个堆，但 G1 的评估机制仍然受 `MixedGCLiveThresholdPercent` 限制。当老年代 region **被大对象完全或近乎完全占用时**，GC 会认为这些 region "存活对象比例高" 而**不予回收**。
- **内存碎片化问题**：频繁的大对象分配可能**导致老年代内存碎片化**，即使总空闲内存足够，也可能因无法找到连续的大内存空间而 OOM。

**根本原因总结**：OOM 发生不是因为 GC 不工作，而是因为 G1 的启发式算法误判了这些大对象的价值。系统实际上有足够的内存，但：

1. 大对象过早进入老年代。
2. GC 误认为这些 region 有价值（存活比例高）。
3. 可用内存被 "僵尸" region 占据。

**所以在测试环境进行一下压测，验证下猜想**。

为了方便触发 OOM 的情况，测试环境机器配置 `-XMS=700M`。通过 JMeter 进行压缩图片的压力测试，单机 QPS 到 10+。测试结果是在 30S 内就会发生 OOM，因为这里 region 大小只有几百 K，**基本都会进入到老年代。测试环境发生 OOM 后查看 GC 日志，发现发生了一次 Full GC**。

**处理方法一：**

优化思路将这些**大对象尽量别分配到老年代**，而是在年轻代被清理掉，所以**通过参数调整一下 region 的大小以及年轻代的大小**。

```
-XX:G1HeapRegionSize=8M
-XX:SurvivorRatio=4
-XX:G1MixedGCLiveThresholdPercent=95(暂时没调整)
```

调整参数后，再次压测就未发生 OOM。

**处理方法二：**

参考 solar-thor 项目的做法，**使用内存池，减少内存碎片的发生**。

#### 容器被杀掉的问题

背景与现象：发布系统中需要**做 APP 差分**，一开始内存较小。**现象是生成差分包的过程中发现服务容器突然消失了**。

原因：**生成差分包会产生大量中间文件，占用大量的内存**，比如两个 200M 的 APP 生成差分，可能需要 `200 * 16` 倍的内存空间。Linux 中 `Memory Cgroup` 可以控制每个容器的最大可用内存。如果进程使用的实际内存大于设置的值，可能进程就会被直接杀掉了（OOM Kill），导致容器直接消失了。

            .category(METRIC_CATEGORY)
            .subsystem(METRIC_SUBSYSTEM)
            .name("notify_msg_intercept_time")
            .help("触达消息拦截处理时间统计")
            .labelNames(new String[]{"notifyType", "templateKey", "templateId", "userIdType", "userId", "result", "eventType"})         //标签中记录了userId

                                                        EnumNotifyEventType notifyEventType
    ) {
        long delayTime = System.currentTimeMillis() - messageContent.getRequestTime();
        ExportMetricUtils.histogram(NOTIFY_MSG_RESULT_HISTOGRAM, delayTime,
                messageContent.getNotifyType().toString(),
                StringUtils.defaultString(messageContent.getTemplateKey()),
                String.valueOf(messageContent.getTemplateId()),

**为什么高基数标签会导致 CPU 占用高？**

这种情况称为**高基数标签**。

- **指标膨胀**：Prometheus 中**每个唯一的标签组合会创建一个新的时间序列（Time Series）**。例如：`api_requests_total{userId="123", path="/foo"}` 和 `api_requests_total{userId="456", path="/foo"}` 是**两个独立的时间序列**。**如果 userId 是动态值（如用户 ID、SessionID、IP 等），\*\***时间序列数量会爆炸式增长（高基数标签问题）\*\*。
- **内存和计算压力**：应用侧：Micrometer/Prometheus 客户端**需要维护大量时间序列的实时状态**（如计数器、直方图），占用大量内存和 CPU。Prometheus 服务端也会在存储和查询高基数指标时，性能急剧下降。
- **垃圾回收（GC）压力**：频繁**创建和销毁时间序列对象会导致 JVM 频繁触发 GC，进一步推高 CPU 使用率**。

可见这里这个**线程池的核心线程数配置为 2000，这 2000 个线程是一直存活的**，但从监控来看，实际上根本用不到这么多线程。

修改验证结果：修改核心线程数为 120。

```java
@EnableAsync
@Configuration
public class ThreadPoolConfiguration {

    private final Logger LOG = LoggerFactory.getLogger(getClass());

    @DynamicConfig(name = "filterExecutorSize", defaultConfig = "1000")
    private int filterExecutorSize;

    // 核心线程数为100
    @DynamicConfig(name = "filterExecutorCoreSize", defaultConfig = "100")
    private int filterExecutorCoreSize;

    @Bean("filterExecutor")
    public Executor filterExecutor() {
        LOG.info("filterExecutor filterExecutorSize = [{}]", filterExecutorSize);
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        // TODO 压测
        executor.setThreadNamePrefix("filterExecutor-");
        executor.setCorePoolSize(filterExecutorCoreSize);
        executor.setMaxPoolSize(filterExecutorSize);
        executor.setQueueCapacity(0);
        executor.setKeepAliveSeconds(5);
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.AbortPolicy());

        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(30);
        executor.initialize();

        return executor;
    }
}
```

![](/zimg/11.故障处理与问题优化-1775399544478.png)

**结论**：

1. `orion-experiment` 项目中打印实验相关日志到本地文件时，**Log4j 配置 includeLocation 为 true，表示需要记录调用位置源码行号等信息，但在日志的 Pattern 中，并没有打印源码行号等信息，所以可以将其关掉，减少 CPU 占用**。
2. **代码中线程池配置不合理导致空闲线程数过多，导致 prometheus 打点获取所有线程信息时耗时过长。需要将核心线程数修改为合理值**。

### 问题排查技巧

#### ⭐ 容器 Load 飙高问题排查过程

**每次应用在发布过程中，刚刚重启好的机器经常会有 CPU 利用率和 load 飙高的现象**，进而导致应用的 RT 变高，很多调用方反馈有大量超时。

![](/zimg/11.故障处理与问题优化-1775399578984.png)

**应用刚刚发布后的几分钟内**，**CPU 飙高到 70%**，Load 飙高到 11，并持续几分钟后就好了。

**问题定位：这个问题的排查过程挺复杂的，刚开始怀疑是应用代码**的问题，但是经过多方检查，打印各种堆、栈的 dump，分析各种火焰图，都没有看到有什么异常。后来又怀疑虚拟机、容器、宿主机是不是有问题，前后换了机器配置、Docker 镜像都没有什么变化。后来又怀疑 JDK 的版本、堆内存设置、垃圾收集器等可能有关？但是排查下来也都发现并没有什么特别的问题。

最后经过多方讨论，定位到和 **JIT 优化有关**。Java 代码需要**先通过 javac 编译成 class 文件**，然后再通过**解释器将字节码将其翻译成对应的机器指令逐条读入，逐条解释翻译执行**。但这个过程太慢了，于是会进行 JIT 优化，JIT 优化器会**基于热点代码检测**，把热点代码直接翻译成机器语言，方便后续直接执行。大大提升了效率。JIT 优化是在运行期进行的，并且也不是 Java 进程刚一启动就能优化的，是需要先执行一段时间的，因为它需要先知道哪些是热点代码。所以在 JIT 优化开始之前，所有请求都是要经过解释执行的，这个过程就会相对慢一些。

而且如果应用的请求量比较大的的话，这种问题就会**更加明显**，在应用启动过程中，会有大量的请求过来，这就会导致解释器持续的在努力工作。**一旦解释器对 CPU 资源占用比较大的话，就会导致 CPU、LOAD 等飙高，导致应用的性能进一步下降。这也是为什么很多应用在发布过程中，会出现刚刚重启好的应用会发生大量的超时问题了。**

而随着请求的不断增多，JIT 优化就会被触发，使得后续热点请求的执行可能就不需要在通过解释执行了，直接运行 JIT 优化后缓存的机器码就行了。

解决思路：**蓝绿部署，进行服务预热**。

#### 服务器磁盘满了该如何处理？

当线上服务器磁盘满了会导致服务不可用、数据无法写入甚至系统崩溃。首先要做的就是**尽快把空间释放**了，避免情况进一步恶化。首先是登陆到机器上（如果都无法登录了，那就只能**置换机器**了。）

**查看磁盘使用情况**

```
df -h
du -sh /*
```

**找出哪个分区、哪个目录占用了最多空间**。

`df`（disk free）用于显示文件系统的磁盘空间占用情况，包括每个挂载点的总空间、已用、可用和挂载路径。

```
df -h

Filesystem      Size  Used Avail Use% Mounted on
/dev/sda1        40G   30G  8.0G  80% /
tmpfs           1.9G     0  1.9G   0% /dev/shm
```

`du`（disk usage）用于显示指定目录或文件占用了多少磁盘空间。常用于查找哪个目录或文件占用空间大。

```
du -sh /*

2.1G    /home
1.2G    /var
300M    /usr
```

`du -h --max-depth=1 /var` 显示 `/var` 下一级目录占用空间大小。

\*\*清理临时文件。清空系统临时文件：

```
rm -rf /tmp/*
```

**清理日志文件：多数情况下，磁盘满了都是**日志打印的太多\*\*了，这时候可以先到应用日志目录下，看一下这些文件的大小，对于一些历史日志文件，如果已经完成 ELK 的采集和同步了，就可以直接 `rm -rf` 物理删除了。

如果日志只有一个文件且特别大，**大概率是 logback 没有配置对**，导致日志没有分成多个文件，这时候根据情况，不能直接直接删除文件，因为应用还在用，这时候可以考虑 `> file_name` 或者 `echo "">file_name` 的方式让日志内容清空。但是可能会导致日志丢失。

#### 如何排查网络的问题？

可以使用一些**基本的命令**检测是否出现丢包、目标端口异常：

- `ping` 检测本机到**目标主机**是否通畅。
- `telnet` 检测本机到**目标主机端口**是否通畅。
- `traceroute` 检测本机到**目标主机经过的路由**是否通畅。
