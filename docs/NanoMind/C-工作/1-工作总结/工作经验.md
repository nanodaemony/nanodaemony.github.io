## 经验

- 选用技术，需要进行调研。看看大厂怎么做的？有哪些通用实现。自己的业务怎么选。
- 谈到优化的几个方向：1. 能不能想办法减少问题？比如请求过多，就把请求减少。2. 使用的框架本身有何优化点？3. 操作系统相关有何优化点。4. 业务是否能优化？比如简化业务流程。
- 产品初期重点做体验，所以需要优化各种不舒服的点，许多功能都是采用原生开发。后期是做运营和维护，所以需要更加灵活进行产品配置，这时候可以考虑大前端开发。参考小猿口算与学练机项目的发展。
- 基础的通用的能力，要减少耦合业务逻辑，不然其它的业务就很难接入进来。比如学习机的配置首页，通用的与业务无关的配置流程才是应该抽象出来的东西，视频课这种有很多业务逻辑，不应该放到 `megrez-config` 的通用逻辑中，而应该放到类似网关层的地方。服务的边界一定要定义清楚。哪怕通用的能力很薄也没关系，一旦耦合了业务逻辑，其它的业务就需要理解才能接入进来了。
- `Program = Logic + Control + DataStructure`。三者之间相互隔离好，代码就更容易维护。
- 写方案的时候可以**列出几个方案，进行对比选型**，更有说服力。
- 需求能砍掉就砍掉，砍不掉的想办法砍掉。
- 问题一定有提前暴露，提前想好，不要等到最后再发现问题。

## 开发

- 能用 `private` 就用，不用轻易给 `public` 方法，不然别人都可能随便用，导致一些意想不到的问题，且难以维护。
- 方法参数过多时，可以考虑抽成一个对象。
- Thrift 里面不是必须的字段一定要设置成 `optional`，不然其 `isSet()` 方法会默认为 true。巨坑。
- 学习技术不用去记具体的参数，配置等各种杂乱的东西。更应该去理解的是该技术是如何发展，如何设计的。为什么要采取这种方案实现。理解原理比会配置更重要。
- 修改东西的逻辑，尽量维护在一个端，不然维护成本很高，改动涉及的端也多。
- 对线上环境要有敬畏之心。 --- 李建红

## 设计

- 不要过度设计，大道至简。简单的设计往往更好。
- 先设计好再开发，不然容易返工。一定要解决风险再开发。
- 用户涉密的信息不要明文存数据库。
- 对于排行榜之类的需要计算超过多少人的需求，需要考虑数据量的增长给计算带来的复杂度。比如随着业务增长，可能数据量会上亿，这时候做统计会很慢。可以考虑的方案：1. 只取部分数据统计，比如最近一个月。2. 定期跑一个大概的百分位占比的汇总表，这样可以免去查询 DB 的操作。
- 先考虑业务架构，再考虑技术架构。
- 新技术的引入要慎重，需要考虑是否对线上有影响。
- 设计数据库表时，一定要考虑一下这个表未来数据的增长量，如果很多的话，一定要提前进行分库分表。
- 遇到问题时，可以跳出或者推翻原有的设定，可以从更高层，或者更底层看看有没有解决方案。
- 对于金额相关的需求，可靠性和稳定性是最需要保证的，可读性和拓展性相对可以次要一些。特别是对于大促类的营销活动，需求变化也非常快，保证需求完成是非常重要的。

## 沟通

- 开会一定要提前发起通知，并告知时间，地点，主要内容，如果有资料可以提前给到。
- 开会或者讨论一定要有结论，对于每次讨论都需要进行总结。如果有时间安排，需要确定一个时间进度。让领导知道自己做的事的进度，不要让领导来主动问进度。
- 开完会之后，可以在群里面同步一下开会的结论，周知所有人。
- 有什么问题、改动、进展、上线等，一定要在群里面说出来，周知大家。
- 要学会砍需求。不是一有需求就要做。要有自己的想法。
- 有风险、有待办一定要即使同步，信息互通非常重要。不同步出来自己可能会忘掉。
