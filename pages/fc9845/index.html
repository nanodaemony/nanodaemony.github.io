    # 故障复盘(处理过程&现象)

    20:00 开始企业微信有一些leo-exam的报警信息

    20:48 定位到异常原因是数据库的主从延迟

    通过企业微信报警错误内容来看, 首先猜测是查询时未查出数据. 然而这部分业务在设计时, 是保证数据库里有数据时才会请求这个接口进行查询, 所以将异常定位在数据库的范围中.

    观察监控, leo-c-mysql 集群的主从延迟比较高, 已经接近 30s, 所以导致数据插入后, 查询的接口查询不到记录, 导致 NPE.

    20:56 DBA排查主从延迟比较高的原因

    DBA 反馈是由于主库写入比较高, 同时从库的读压力比较大, 导致主从延迟增加. 比较高的读是来自于 leo-midas-group-activity, 该服务只配置了一个从库, 所以导致单个库的流量比较大.

    从整体集群上看, 从库的负载不均衡, 大部分请求都打到了其中一个从库上.

    21:00 主从延迟恢复正常, 服务恢复

    高峰期已过, 流量慢慢降下来, 主从延迟在 21:00 恢复到正常, 报警信息也没有了, 服务错误 QPS 也恢复正常.

    # 故障原因

    leo-c-mysql 集群的写流量比较大, 同时一个从库的读流量比较大, 导致主从延迟增加.

    # 优化

    # 1.增加从库

    流量较大的服务(leo-midas-group-activity)增加多从库配置, 减小单个从库压力

    # 2.原因排查

    06-13 号发生了比较长的主从延迟, 后续观察 06-14 仍然在相同的时间点有较高的主从延迟, 需要排查一下问题原因.

    观察 MySQL 集群的监控, 在 06-07 开始后的几天, 写 QPS 上涨了很多.

    统计写请求最高的库

    sum (rate(mysql_perf_schema_table_io_waits_total{ instance=~".*leo-c-mysql.*", operation=~"update|delete|insert"}[30s]) ) by (instance, schema)
    1

    统计是哪张表的写请求

    sum (rate(mysql_perf_schema_table_io_waits_total{ instance=~".*leo-c-mysql.*", operation=~"update|delete|insert", schema="leo_exam"}[30s]) ) by (instance, schema, name)
    1

    观察服务概览中写 MySQL 的请求也是能够对的上的

    排查代码, 找到流量来源

    排查 leo-exam 代码, 这张表的写入有两个来源, 一个来源是拍照检查的 MQ, 一个来源是 addAnimationKeypointVideos RPC 接口. 看接口监控, addAnimationKeypointVideos 接口的监控比较正常, 但是拍照检查 MQ 的 QPS 和数据库 QPS 上涨规律基本一致. 但是峰值 QPS 相差 3 倍, 看业务逻辑是因为拍照检查一般一次最多 3 个题目, 拿到题目之后会去内容组的接口查询有无视频, 然后 Insert 或者 update search_question_video 这张表, 所以 QPS 相差三倍.

    看拍照检查的 QPS 趋势基本一致

    优化了 SQL:

    • leo-exam 中 insert 可以批量写
    • leo-exam 中先 select 后 update/select 可以改为一次 insert into ... on duplicate key update ...
